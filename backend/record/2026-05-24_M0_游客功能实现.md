## 1. 目标

实现 API 文档 §1「M0 游客 / 演示模式」要求的后端能力：

1. **POST /api/auth/guest-login** — 无需鉴权，POST 即返回 JWT；
2. 游客 Access Token 有效期 **2 小时**，Refresh Token 有效期 **24 小时**；
3. 返回体含 `is_guest: true` 与 `quota: 2`（每会话视频生成上限 2 条）；
4. 游客限制：
   - ❌ 不可修改密码
   - ❌ 不可删除预置演示数据
   - ❌ 每次会话最多触发 2 次视频生成；
5. 统一响应格式 `{code, msg, total, data, traceId}` 落地。

## 3. 关键实现细节

### 3.1 游客身份识别

按 DB 文档 §2.1 的设计，**不引入独立游客表**，复用既有 `users` 表的预置行：

- ID：`a0000000-0000-0000-0000-000000000001`
- 邮箱：`demo@vidcraft.io`
- 昵称：`体验用户`

`is_guest` 通过 ID 常量比较得出（`isGuestId()`），保持表结构纯净。

### 3.2 Token 签发

```ts
const accessToken = await jwtService.signAsync(
  { sub: guest.id, email: guest.email, is_guest: true, session_id },
  { secret, expiresIn: 7200 },   // 2h
);
const refreshToken = `rt-guest-${randomBytes(16).toString('hex')}`;
await redis.set(`auth:refresh:${refreshToken}`, JSON.stringify({...}), 86400); // 24h
```

- **Access Token**：JWT payload 嵌入 `is_guest`、`session_id`，`JwtStrategy.validate()` 解出后挂到 `req.user`；
- **Refresh Token**：自定义 `rt-guest-*` 前缀字符串，写入 Redis 以便后续 `/api/auth/refresh`、`/api/auth/logout` 校验/吊销。

### 3.3 会话配额（每会话 2 次视频生成）

- `guestLogin()` 时按 `session_id` 初始化 Redis 键 `auth:guest:quota:{session_id} = 2`，TTL 与 Refresh Token 同步（24h）；
- 视频生成接口接入 `consumeGuestQuota(sessionId)` —— Redis `DECR` 原子操作，结果 < 0 时回滚 + 抛 `403`；
- 不写库（DB 文档 §2.1 备注：「游客固定 2/会话 由业务层兜底，不写库」）。

### 3.4 游客限制（修改密码、删除预置数据等）

提供两条等价 API：

1. **声明式**：在受限接口加 `@UseGuards(JwtAuthGuard, BlockGuestGuard)`；接口默认不让游客通过，加 `@AllowGuest()` 放行；
2. **命令式**：在 Service 内调用 `authService.assertNotGuest(userId, '修改密码')` 抛 403。

### 3.5 统一响应格式

文档 §0.3 要求所有接口返回：

```json
{ "code": 200, "msg": null, "total": 0, "data": {...}, "traceId": "..." }
```

`TransformInterceptor` 自动包裹 Controller 返回值；若需要自定义 `total`、`msg`、`code`，使用 `envelope(data, { total, msg })` 包装即可。

`HttpExceptionFilter` 同步使用相同结构，HTTP 状态码与 body `code` 字段保持一致。

### 请求

```http

```

### 内部流程

```
Controller.guestLogin()
  → AuthService.guestLogin()
     ├─ ensureGuestUser()        // 幂等查或建 demo 用户
     ├─ signAccessToken()        // JWT, exp=2h, claims={sub, email, is_guest, session_id}
     ├─ storeRefreshToken()      // Redis SET ttl=86400
     └─ initGuestSessionQuota()  // Redis SET quota=2 ttl=86400
```

## 8. 风险与注意

1. **演示用户密码 hash**：迁移文件中使用 bcrypt 占位 hash（`$2b$10$N9qo8uLOickgx2ZMRZoMye...`），仅用于满足 `password_hash NOT NULL` 约束；游客登录走 JWT 直发，**不走密码校验**，故占位 hash 不影响功能；正式账号登录时若误用 `demo@vidcraft.io` 邮箱不会通过校验；
2. **Redis 依赖**：若 Redis 不可用，guest-login 会抛 500；后续可加降级（in-memory Map），目前与 BullMQ 共享 Redis，可用性同步；
3. **session_id 与 Refresh Token 解耦**：当前 Refresh Token 与 session_id 是两条独立 Redis 键，refresh 时需重新签发同 session_id 的 Access Token 以共享配额；该逻辑留给 `/api/auth/refresh` 实现时处理。
