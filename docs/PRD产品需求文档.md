# 一、业务背景与问题分析

## 1.1 市场背景

TikTok Shop 已成为全球电商增量市场的重要阵地，短视频是商家获取流量与提升成交的核心载体。然而商家在视频内容生产上面临多重痛点：

|  |  |  |
| --- | --- | --- |
| **痛点维度** | **现状描述** | **量化影响** |
| 制作成本高 | 传统实拍单条视频成本 2000–5000 元 | 中小商家难以持续产出 |
| 制作周期长 | 实拍 + 剪辑周期 3–7 天 | 无法快速跟上热点节奏 |
| 内容同质化 | 现有一键AI生成工具输出雷同 | 平台流量分发受限 |
| 创作是黑盒 | 无法对视频分镜、风格进行干预 | 商家缺乏对内容的掌控感 |
| 数据孤岛 | 发布后数据无法回流指导创作 | 无法持续优化转化效果 |

## 1.2 用户画像

|  |  |  |
| --- | --- | --- |
| **用户类型** | **规模特征** | **核心诉求** |
| 中小卖家（主力） | TikTok Shop 中腰部商家，月GMV 5–50万 | 快速低成本出片；操作简单，不需技术背景；一键完成从商品到视频全流程 |
| MCN机构运营 | 多账号、多品类管理 | 批量生产视频；团队协作支持；精细化数据归因指导选品方向 |
| 品牌方内容团队 | 有一定制作能力的专业用户 | 精细化分镜控制（含因子替换）；保持品牌调性一致性；支持多平台格式导出 |

## 1.3 核心业务价值

|  |
| --- |
| **一句话定位**  VidCraft 是面向 TikTok 电商商家的 AIGC 视频工作站，通过「优质视频库 → 剧本生成（含因子干预）→ 视频创作」完整闭环，将出片周期从 3–7 天压缩至 30 分钟，让每条视频都比上一条卖得更好。 |

## 2.2 核心功能模块详述

### 2.2.0 游客/演示模式（P0，新增）

目标：确保评委无需注册即可在最短路径内体验完整系统价值，符合赛题「可访问性」原则。

* 演示环境首页提供「游客体验」一键入口，自动使用预置演示账号登录，无需输入任何信息。
* 预置内容：示例项目 × 2（含已解析商品信息）、已生成剧本 × 3（含不同风格和因子替换示例）、已生成视频 × 1（含 Mock 数据看板和预置诊断报告）。
* 游客账号可新建项目、上传素材、生成剧本、提交视频生成任务（配额限制 2 条/会话）。
* README 明确说明演示地址 + 游客入口位置 + 预置内容清单。

### 2.2.1 素材模块（P0）

目标：为AIGC视频生成提供高质量、可检索的结构化素材资产库。

* 素材入库：支持商家手动上传商品主图、商品视频、参考素材等多类型文件（图片/视频）。
* 多颗粒度结构化：NestJS VolcanoApiService 统一调用 Doubao Vision + Doubao Embedding（1024维）进行多模态理解和向量化；视频素材由 NestJS 内联 FFmpeg/ffprobe 取时长并抽取关键帧（≤30s）后打标（所有 AI 调用统一经 NestJS）。
* 素材检索：提供基于关键词、标签、向量相似度（1024维余弦相似度）的多颗粒度检索，支持剧本/创作模块的智能召回。

### 2.2.2 剧本模块（P0/P1）

目标：自动化生产优质电商AIGC视频剧本，参考「找参考 → 提炼方法论 → 生产剧本」三步链路，沉淀优质视频库 / 模板 / 剧本三类核心实体。

**优质视频库（P0，v1.0 缺口补全）：**

* 功能定位：按类目/关键词检索站外（YouTube/TikTok/Instagram 等）爆款视频，每条视频配结构化拆解报告（Hook手法、卖点、分镜节奏、风格等），支持上传自有视频做拆解。
* URL 导入链路：商家粘贴公开视频 URL → 后端 yt-dlp 获取公开元数据（标题/描述/封面图）→ Doubao Vision 分析封面图 + Doubao 大模型生成结构化拆解报告 → 入库 viral\_library 表。
* 自有视频拆解链路：上传本地视频 → FFmpeg 切帧 → Doubao Vision 分析关键帧 → 汇总生成拆解报告 → 入库。
* 合规约束：仅保存公开视频的结构化分析结果，不复刻、不存储原始视频本体；每条条目声明来源 URL 与平台；作业声明素材来源。

**方法论提炼（灵感模板）：**

对同品类爆款视频做聚类（n:1），归纳为结构化创作模板，包含策略（视频创作的抽象方法）+ 因子（视频创作的具体手段）。因子维度包括：视觉风格、开场手法、旁白风格、节奏密度、CTA形式。

**剧本生成：**

输入商品ID/信息（标题、卖点、目标人群、场景），由「策略 + 因子 + 商品信息 + 约束规则」组合生成完整剧本，包含叙事框架、视觉风格、分镜脚本（含画面描述、镜头运动、BGM、字幕）、约束清单。

**多种生产模式：**

* 爆款仿写：自动匹配最佳爆款（从优质视频库或爆款基因库中选择）或人工指定参考，融合商品信息生成同款剧本。
* 灵感模板：商家挑选模板 + 商品ID → 融合出剧本。
* 剧本自动化（进阶）：分层解耦「策略库+因子库+约束库」，动态组合，每次产出多套风格各异剧本。

**剧本干预（含因子局部替换，v1.0 问题2补全）：**

* Prompt 微调：直接编辑单个分镜的画面描述、配音文案。
* 分镜增删：插入新分镜或删除多余分镜。
* 台词改写：修改配音文案后局部重生成。
* 因子局部替换（新增）：通过右侧「创作因子」面板选择替换某个维度的值（如将视觉风格从「黑风极简」改为「夏日度假风」），系统根据新因子调用 Director Agent 重生成受影响分镜，未受影响分镜保留。支持操作历史（因子替换记录 + 撤销）。

### 2.2.3 创作模块（P0/P1）

目标：根据素材和剧本进行视频创作，视频时长不超过15秒。

* 一键成片：仅需输入商品链接或上传商品主图，端到端产出完整带货视频。
* 智能剪辑：基于素材切片三层标签体系（商品/视频/slice），结合剧本分镜，自动完成画面拼接、转场、字幕、配音（支持中/英/日/韩多语种）、配乐合成。
* 分镜级干预：支持用户对单个分镜重生成、替换素材切片、调整时长、修改台词；干预后整体视频快速重渲染。
* 预览与导出：提供在线预览，支持导出多种分辨率与画幅（竖版9:16、横版16:9等），适配不同分发渠道。

# 三、非功能性需求

## 3.1 性能需求

注：标注「估算」的指标为基于技术选型的经验性估计，「待Week4压测」的指标将在 Week 4 实测后更新文档，不以未验证数字作为承诺（v1.0 问题10修订）。

|  |  |  |
| --- | --- | --- |
| **指标** | **目标值** | **说明与依据** |
| 脚本生成响应 | < 10 秒 | LLM首token到完整脚本输出；基于 Doubao-Seed-2.0-pro 流式输出，估算值 |
| 任务进度刷新间隔 | ≤ 5 秒 | 前端 HTTP 轮询 GET /status，建议间隔 5s |
| 视频合成时间（≤15s视频） | 目标≤60秒，待Week4压测 | FFmpeg CPU合成（拼接+字幕+BGM+配音），实际时间与服务器配置（CPU核数/内存）强相关；1080P可能超60秒，720P更可控；Week 4 在实际部署环境压测后更新此值 |
| 页面首屏加载 | < 3 秒 | Vite构建 + 代码分割 + Nginx gzip，估算值 |
| 素材向量检索延迟 | < 500ms | pgvector HNSW 索引，1024维，数据量 < 10万条时，估算值 |

## 3.2 可靠性需求

* 视频生成任务失败自动重试，最多 3 次，超时设置 15 分钟/任务。
* 任务状态持久化至数据库（含 trace\_id），页面刷新后任务进度不丢失。
* 生成失败需有清晰的错误反馈（失败原因、traceId 便于排查、重试按钮）。
* Seedance API 回调超时兜底：超过20分钟未收到回调，自动轮询补偿。
* 生成过程链路追踪：OpenTelemetry 覆盖 NestJS→Seedance回调 链路，traceId 随任务传播，便于排查长链路任务问题（P1 要求的「生成过程trace」）。

## 3.3 安全性需求

* JWT 无状态鉴权，Token 有效期 7 天，支持刷新。
* 火山引擎 API Key 仅存储于服务端环境变量，统一经 VolcanoApiService 调用，严禁暴露至前端。
* Seedance 回调端点安全：采用 HMAC-SHA256 签名验证（创建任务时在 callback\_url 携带签名 token），回调接收时验证签名，防止伪造请求篡改任务状态。
* 用户上传文件做类型校验（MIME类型 + 文件头校验），防止恶意上传。
* 用户视频文件按用户ID隔离存储，禁止越权访问。
* API 接口限流：每用户每分钟最多触发 5 次视频生成任务。

## 3.4 扩展性需求

* 异步任务支持水平扩展（Bull 多 worker 消费同一 Redis 队列）。
* 模型 API 可插拔：通过 Adapter 模式抽象 LLM 调用层（VolcanoApiService），支持切换不同模型提供商，不影响业务逻辑。
* 模块间通过清晰 Interface 解耦，「优质视频库—基因库—剧本—创作」各模块独立演进。
* Embedding 维度：Doubao Embedding 输出 1024 维，数据库 schema 使用 vector(1024)，HNSW 索引在初始化时创建，切换模型时需重建索引（有文档说明）。

## 3.5 用户体验需求

|  |  |
| --- | --- |
| **体验维度** | **具体要求** |
| 访问门槛 | 提供游客模式，评委无需注册即可体验核心功能；预置演示数据确保核心链路可完整演示 |
| 长任务体验 | 设计合理进度展示（分镜级进度条）、断点续传、异常重试与失败兜底（含traceId便于排查） |
| 分镜编辑器 | 分镜列表+时间轴复杂交互：支持拖拽、快捷键、实时预览；单个分镜局部刷新不重渲染整片；右侧因子面板支持一键替换风格 |

# 四、约束与合规要求

## 4.1 技术约束

* 前端：React 19 + TypeScript（赛题要求）。
* 后端：Node.js + TypeScript（赛题要求），单一 NestJS 服务承载业务逻辑与 FFmpeg 内联合成。
* AI 能力：优先接入火山引擎 OpenAPI（Doubao大模型、Seedance视频生成、TTS、Embedding），统一经 NestJS VolcanoApiService 调用。
* 代码规范：集成 ESLint、Prettier、Husky、StyleLint，统一代码风格。
* 仓库管理：单一 Git 仓库（Monorepo），前后端代码统一管理。

## 4.2 素材合规约束

* 优质视频库只保存公开视频的结构化分析结果，不复刻、不混剪原视频。
* 所有参考素材必须声明来源（source\_url + platform + declared\_at），UI层展示来源声明。
* yt-dlp 仅用于获取公开视频元数据（标题/描述/封面图），不下载、不存储原始视频本体。
* 演示使用公开可商用素材或自有素材，设计「合规声明 + 真实性校验」工作流。

## 4.3 数据说明

分发与转化数据使用 Mock 数据或自建模拟服务。取舍理由：TikTok 数据授权流程复杂，会阻塞演示；使用 Mock 数据可实现完整的数据飞轮逻辑（指标展示→AI诊断→优化建议→分镜编辑→重生成），重点在归因算法设计、可视化呈现质量与数据结构合理性，而非数据接入本身，符合赛题「数据可使用mock数据或自建模拟服务」的说明。

Mock 数据设计原则：预设 3–5 组典型场景（高转化/低完播/低转化/正常），数据结构与真实场景完全对齐，归因逻辑完整实现，评委可以通过演示完整体验数据飞轮的价值。

# 五、可行性分析

## 5.1 技术可行性

* 火山引擎已提供 Doubao-Seed-2.0-pro（文本大模型）、Doubao-Seedance-1.5-pro（视频生成）、Doubao Embedding（1024维）API，可直接接入。
* FFmpeg 视频合成方案成熟，支持字幕/BGM/配音/转场全链路合成。
* yt-dlp 开源工具可获取公开视频元数据，实现优质视频库导入链路。
* React + NestJS + Bull + PostgreSQL + Redis 技术栈成熟，全栈团队可快速落地。
* OpenTelemetry SDK 开箱即用，NestJS 有成熟接入方案。
* 向量检索采用 pgvector（1024维 HNSW 索引）降低部署复杂度，满足演示需求。
